## Цель работы

Сравнительный анализ времени выполнения основных операций стека (push, pop) при различных сценариях нагрузки для определения наиболее эффективной реализации в зависимости от условий эксплуатации и объема данных.

---

## Структура проекта

* `arr_stack.h / arr_stack.c` — реализация стека на основе динамического массива.
* `list_stack.h / list_stack.c` — реализация стека на основе односвязного списка.
* `tests_arr.c / tests_list.c` — реализация тестов на массиве и списке.
* `[serial number]_tests.txt` — текстовые файлы с результатами экспериментов.
* `laba.py` — файл построения графика на основе результатов эксперимента.
* `plot_test_[num]` — графики построенные прошлым файлом.
* `run.sh` — bash скрипт запуска лабораторной работы.


---

## Реализации

### 1. Стек на массиве (arr_stack)

* **Преимущества:** Высокая скорость доступа к элементам, минимальное потребление памяти (отсутствие указателей связи), эффективное использование кэша процессора за счет последовательного расположения данных.
* **Недостатки:** Накладные расходы на перевыделение памяти при достижении лимита емкости.

### 2. Стек на списке (list_stack)

* **Преимущества:** Отсутствие необходимости в выделении непрерывного блока памяти, стабильное время добавления элемента (не требует периодического копирования всего массива).
* **Недостатки:** Повышенное потребление памяти (хранение указателя `next` в каждом узле), медленная работа из-за частой аллокации и фрагментации памяти.

## Тестирование

Программа выполняет серию тестов для количества элементов $N$ в диапазоне от 1 000 до 1 000 000. При этом, для каждого $N$ тест прогоняется по три раза и полученные результаты усредняются.

1. **Test 1 (Смешанная нагрузка):** Итеративное удаление половины элементов и вставка четверти до момента, когда объем данных станет менее 10% от исходного.
2. **Test 2 (Блочные операции):** Массовое удаление и вставка данных блоками $N$/100 элементов. Позволяет оценить работу аллокатора при резких изменениях размера.
3. **Test 3 (Рандомизированный тест):** Случайная последовательность операций push и pop, имитирующая непредсказуемую пользовательскую нагрузку.
4. **Test 4 (Линейное заполнение):** Заполнение стека с нуля до $N$ элементов для оценки чистой скорости вставки.

## Запуск

```bash
./run.sh
```

После запуска: компилируются файлы с реализацией стека -> запускаются тесты -> генерируются файлы с результатами -> по ним строятся графики.

## Итог

На основе проведенных экспериментов можно сделать вывод, что реализация на базе динамического массива превосходит односвязный список по скорости выполнения операций во всех протестированных сценариях. Основной выигрыш в производительности достигается за счет последовательного расположения данных в памяти и отсутствия накладных расходов на частые вызовы аллокатора (calloc/free).

Тем не менее, стек на списке остается актуальным в условиях сильной фрагментации памяти, когда в системе отсутствуют крупные непрерывные блоки, необходимые для расширения массива.

Итоговый выбор: если критична скорость и есть возможность выделить сплошной участок памяти — следует использовать динамический массив. Если же важнее гибкость распределения памяти в условиях ее дефицита — выбирается связный список.
